# 控制反馈：当前计划与 Defense

这页合并当前计划与当前 defense，避免在执行层重复维护两套相近表述。

> [!important] 执行边界
> 当前阶段只推进两条可输命题：[[局部状态更新闭包]] 与 [[事件对象实验计划|事件对象 -> 可归因 -> 可纠偏]]。任务成功率、完整 test-time training、能力边界和基础模型接口都不是第一击。

## 当前不主张

当前不主张：

- 控制反馈理论已经成立。
- `workspace + load/store` 已经全面优于最强 LLM+Agent。
- 它已经构成独立基础模型接口。
- 它已经自然超出 agent engineering。
- 任务成功率提升已经被证明。

## 当前主张

当前主张的是：

- 长期理论动机是严肃的，包括计算复杂度、`RAM/RASP` vs `write-once/append-only`、test-time scaling、CoT 与时序依赖。
- 现实系统没有消化掉全部问题。最强 LLM+Agent 虽已部分模拟 load/store，并吸收 reflection、planning、tool use、search 等机制族，但仍未说明哪一层是独立工作方式层。
- 因而值得把问题压成两条可输命题：理论侧的 `局部状态更新闭包`，实验侧的 `事件对象 -> 可归因 -> 可纠偏`。

## 理论计划

第一优先级是 [[局部状态更新闭包]]。当前不再继续扩展问题清单，而要尽快把它压到：

- 一个具体任务族
- 状态空间 `S`
- 局部原语集 `U`
- 全局重解释算子 `G`

这要求 `task-first`，而不是 `definition-first`。先在具体任务族上定义 `S/U/G`，再讨论是否能一般化。

第二优先级是在闭包问题有清晰实例后继续追问：

- 局部修复复杂度 gap
- control-step efficiency
- 搜索成本
- 状态更新代价

## 实验计划

第一阶段目标是验证：

- 显式状态语义是否能把 `address/read/write/commit` 统一成稳定运行时事件对象。
- 这些事件是否被后续步骤稳定利用，而不只是日志标签或诊断展示。
- 这种结构是否提升错误可归因性与局部纠偏能力。

实验 protocol 详见 [[事件对象实验计划]]；Plan-v3 的 challenge / defense 详见 [[Bridge与Plan-v3-Challenge-Defense]]。

主指标：

- 控制动作级错误可归因率。
- 局部纠偏成功率。

强次指标：

- 回放可复现率。
- 同类错误复发率。
- 平均恢复步数。

观察指标：

- 最终任务成功率。
- 状态重建成本。
- 全局重解释触发频率。

## 实验组织

确定性层先固定：

- 状态空间
- 局部原语
- 任务族
- 失败类型

目的不是展示智能，而是钉住机制对象本身是否存在。

智能层再使用现有 LLM/Agent 系统，验证：

- 这种机制是否能被现实预测型控制系统承载。
- 事件对象是否会被模型或系统稳定使用。
- 局部状态读写是否减少全局重解释依赖。

## 当前不作为第一击

当前不把以下方向作为第一主命题：

- 总体任务成功率。
- 大范围 test-time efficiency 曲线。
- 更强能力边界。
- 完整 test-time parameter update。

原因是这些方向太容易被脚手架、prompt、工具链、上下文长度和模型能力污染，容易把机制问题与智能强弱问题混在一起。

## Defense 核心

当前为什么押 `局部状态更新闭包`：

- 它比事件对象可观测性更接近机制层。
- 它比局部修复复杂度 gap 更不容易被工程因素污染。
- 它比单纯可归因性更接近长期工作方式动机。
- 一旦成立，能反过来重写实验与效率问题的解释框架。

当前为什么押 `事件对象 -> 可归因 -> 可纠偏`：

- 如果 `address/read/write/commit` 不能形成稳定运行时事件对象，并被后续步骤利用，这条路线仍更像规整 protocol 或 diagnostic-aware tooling。
- 当前实验不是绕开最强 LLM+Agent，而是借助它们作为智能承载体，检验显式状态语义与控制接口是否形成可分离机制层。

## 失败条件

当前路线应接受这些失败条件：

- 无法为具体任务族定义有判定力的 `S/U/G`。
- 运行时事件对象只是日志标签，不能影响后续决策。
- 可归因与可纠偏无法稳定提升。
- 局部修复成本、控制步数、搜索成本或全局重解释频率没有可解释改善。
- 所谓增量完全可被普通 scaffold / prompt / 工具协议解释。

## 结论

当前 defense 不是“控制反馈已经成立”，而是：这条路线已被压成两个值得押、也能输掉的命题；如果它们站不住，就不应继续扩张叙事；如果它们站住，再由效率桥决定是否值得成为更大的工作方式路线。
