# 控制反馈：理论与 Defense

当前理论押注是 `局部状态更新闭包`。如果想回看为什么最后会收缩到这里，再读前面的 [[#控制反馈：理论桥梁与候选命题|桥梁与候选命题]]。

## 一页版理论地图

当前理论侧不要试图一次证明 `Token = Instruction` 的全部长期价值。更稳的结构是：

| 层级 | 命题角色 | 当前状态 |
| --- | --- | --- |
| Candidate C | [[10-控制反馈-TokenInstruction/实验与机制Defense#控制反馈：事件对象实验计划|事件对象 -> 可归因 -> 可纠偏]] | 更接近实验入口，回答“机制对象能否被观测和利用”。 |
| Candidate A | 局部状态更新闭包 | 当前理论主赌注，回答“局部状态更新是否能形成机制闭包”。 |
| Candidate B | 局部修复成本 / control-step efficiency | 后续效率裁决层，回答“机制是否带来可量化收益”。 |

读者应先带走三个判断：

- 当前首要理论赌注是 `局部状态更新闭包`，不是直接证明长期复杂度优势。
- 局部闭包若不能映射到现实任务和效率桥，就只是形式化游戏。
- 理论 challenge 的作用是收缩命题边界，不是扩大问题清单。

## 控制反馈：理论桥梁与候选命题

这一节只处理一件事：为什么当前主线会收缩到 `局部状态更新闭包` 与 `事件对象 -> 可归因 -> 可纠偏`。

> [!summary] 三个候选桥
> Candidate C 是现实实验入口，Candidate A 是理论主赌注，Candidate B 是后续效率裁决层。三者不是互斥路线，而是证据积累顺序。

### 三类桥

#### 理论合法性桥

回答：

- 为什么 `Token = Instruction` 不是纯类比。
- 为什么 AI 工作方式可以类比到 `RAM / RASP / load-store / addressing`。
- 为什么 `append-only / write-once` 可能天然受限。
- 为什么 test-time computation 的组织方式可能影响能力边界。

notes 中的复杂度、RAM/RASP、write-once、CoT、time scaling 讨论主要属于这一类。它让路线“值得想”。

#### 现实落地桥

回答：

- 在今天的 LLM+Agent 现实里，理论动机还剩什么未被已有工具系统消化。
- 哪个层次的问题仍值得单独研究。
- 为什么不能只说“现有 agent 已经够强”。

当前 bridge layer 文档主要在补这一类。它让路线“值得做”。

#### 机制闭环桥

回答：

- 从 `Token = Instruction` 到系统行为为什么会改变。
- 从显式事件对象到后续决策利用。
- 从可归因到可纠偏。
- 从局部纠偏到 control-step / search-step / efficiency proxy 的改善。

这是当前最缺的一类桥。它让路线“能被证明不是空话”。

### 五个理论问题

表示桥问题：

- `Token = Instruction` 是否只是描述重写，还是引入新的可操作变量。

机制桥问题：

- 什么条件下 instruction-like token 不只是被输出，而是真的形成可复用状态访问机制。

现实映射桥问题：

- 在 agent/tool/memory 已经很强的现实下，`Token = Instruction` 剩余的不可替代增量是什么。

效率桥问题：

- 如果 `Token = Instruction` 重要，它最终改善什么。
- 候选包括局部修复成本、控制步数、搜索成本、状态更新代价、test-time efficiency。

内生化桥问题：

- 什么时候它只是 protocol。
- 什么时候它开始逼近 model interface。
- 什么时候它才算从 agent engineering 往基础工作方式推进。

### Candidate A / B / C

#### Candidate C：事件对象 -> 可归因 -> 可纠偏

对应页面：[[10-控制反馈-TokenInstruction/实验与机制Defense#控制反馈：事件对象实验计划|事件对象实验计划]]

核心命题：

- `workspace + load/store + [[10-控制反馈-TokenInstruction/实验与机制Defense#显式状态语义|显式状态语义]]` 是否能把 `address/read/write/commit` 统一成稳定运行时事件对象。
- 这些对象是否被后续步骤稳定利用。
- 这种结构是否提升错误可归因性与局部纠偏能力。

位置：

- 主要属于现实落地桥。
- 同时向机制闭环桥延伸。

优势：

- 最贴近原始 load/store/address generation 直觉。
- 最容易设计第一批实验。
- 最容易形成对照与消融。

风险：

- 容易退化成 diagnostic-aware protocol。
- 容易停留在“事件被表示出来”，而不是机制真正成形。

#### Candidate A：局部状态更新闭包

对应页面：[[10-控制反馈-TokenInstruction/理论与Defense#局部状态更新闭包|局部状态更新闭包]]

核心问题：

> 一类任务中，系统是否能主要通过有限种局部状态更新原语完成修复与推进，而不必反复做[[10-控制反馈-TokenInstruction/主线-计划-问题地图#全局重解释|全局重解释]]。

它关心：

- 局部状态更新是否形成闭包。
- 后续步骤是否受局部更新约束。
- 局部修复是否能在该闭包内完成。

位置：

- 主要属于机制闭环桥。

优势：

- 比 Candidate C 更接近机制层。
- 更能回应“这不是纯事件表示层”的攻击。

风险：

- 抽象程度高。
- 第一轮实验更难切。
- 若定义不清，容易滑回概念层。

#### Candidate B：局部修复成本 / control-step efficiency

对应页面：[[10-控制反馈-TokenInstruction/主线-计划-问题地图#双命题与 Primary Bet 层级|双命题与Primary-Bet层级]]

核心问题：

> 如果显式状态语义真的重要，它是否降低局部修复所需的控制步数、搜索步数、恢复步数或其他局部成本 proxy。

它关心：

- 局部修复成本。
- control-step 数量。
- 搜索成本。
- 恢复成本。

位置：

- 机制闭环桥与效率桥问题的交界处。

优势：

- 更接近长期效率直觉。
- 更容易与 test-time scaling 讨论相连。

风险：

- 太容易被脚手架、prompt、工具链污染。
- 如果机制层没站稳，成本解释会很脆。

### 三者关系

当前证据顺序是：

1. Candidate C 先把对象变成现实里可观测、可实验的对象。
2. Candidate A 再问它是不是形成真正的局部状态更新机制。
3. Candidate B 最后问这个机制是否带来局部修复或控制成本优势。

这不是理论必然依赖，而是证据积累顺序：

- C 解决“看不见、测不到”。
- A 解决“是不是机制”。
- B 解决“有没有效率收益”。

### 理论计划中的四层结构

问题意识层：

- 回答为什么这条路线值得想。
- 包括 memory/workspace 语义、test-time computation 组织、append-only/write-once 限制。

纯理论命题层：

- 回答哪些对象可以先不靠实验，直接数学形式化。

机制桥层：

- 回答哪些理论对象最适合接到现实实验。
- 包括运行时事件对象、局部状态更新闭包、局部修复成本 proxy、一致性维护代价。

实验计划层：

- 回答当前先测什么。

### 当前优先级

第一优先：[[10-控制反馈-TokenInstruction/理论与Defense#局部状态更新闭包|局部状态更新闭包]]

- 它最接近机制层。
- 能连接长期动机与现实实验。
- 能为 Candidate A 提供理论骨架。

第二优先：局部修复复杂度 gap

- 它最接近长期效率直觉。
- 能为 Candidate B 提供更硬理论方向。

第三优先：控制动作级错误归因的可判定性

- 它直接服务当前 bridge 与实验计划。
- 能帮助把“可归因 + 可纠偏”从工程语言转成理论语言。

### 不适合直接作为第一理论命题

当前不建议第一击直接押：

- `Token = Instruction` 是否全面优于当前 agent 实践。
- 更强能力边界是否已经成立。
- 总体 test-time scaling 曲线是否显著优于主流系统。

这些命题太大，现实系统细节污染太强，理论与实验之间缺少中间层。

## 局部状态更新闭包

这页是控制反馈线当前理论主赌注的形式化草案。目标不是给出最终定义，而是固定对象、问题和候选命题形状。

> [!summary] 理论主赌注
> 对某类现实相关任务，求解与纠错能否由有限局部状态更新原语持续完成，而不必频繁退回全局重解释。

### 直观问题

> 对某类任务，求解与纠错是否可以主要通过有限组局部状态更新原语完成，而不必周期性退回全局重解释？

更原始地说：

> 这类任务，是不是可以一直靠局部改动做下去，而不必经常把整个东西重新看一遍？

### 最小对象

#### 状态空间 `S`

设 `S` 是系统状态空间。一个状态 `s in S` 可以理解为：

- 某个工作区。
- 某个结构化对象。
- 某个可被局部读写的中间表示。

当前不要求 `S` 是具体哪种数据结构，但要求它至少带有某种“局部部分”的概念，例如：

- 块。
- 槽位。
- 节点。
- 区间。

#### 局部更新原语 `U`

设 `U = {u1, ..., uk}` 是有限组局部更新原语。

每个 `ui` 是一个部分函数：

```text
ui : S -> S
```

并满足直观局部性：

- `ui` 只改动状态的一个有限局部部分。
- 其余大部分状态保持不变。

“局部”的严格度量尚未钉死，后续必须在具体任务族中 task-first 地定义。

#### 任务空间 `T`

设 `T` 是任务集合。每个任务 `tau in T` 给定：

- 初始状态 `s0(tau)`。
- 目标条件 `Goal(tau, s)`，表示状态 `s` 是否完成任务。

### 基础定义草案

局部更新可达性：

- 给定任务 `tau`，若存在有限长序列 `ui1, ui2, ..., uim`，使得从 `s0(tau)` 连续应用这些原语后满足 `Goal(tau, s)`，则称 `tau` 在原语集 `U` 下可达。

局部更新闭包：

- 对某类任务族 `T* subset T`，若其中每个任务都在 `U` 下可达，并且求解过程不要求周期性引入某种全局重解释算子 `G`，则称 `T*` 在 `U` 下近似闭包。

### 全局重解释 `G`

`G` 是这个草案中最关键、也最容易含混的对象。

当前工作定义：

- 一个操作如果在决定下一步时，必须重新依赖整个状态的大范围语义解释，而不是只依赖少量局部更新结果，它就更接近 `G`。

`G` 不是实现层概念：

- 不是是否重新 prefill。
- 不是是否重算 KV cache。

`G` 是语义层 / 工作方式层概念：

- 下一步决策是否仍主要依赖对整份全局状态的再次理解。

更现实的行为判据应包括：

- 后续动作是否重新广泛扫描状态。
- 局部更新结果是否足以约束下一步。
- 缺少局部约束时是否退回大范围搜索或解释。

### 候选命题

#### 命题 A：闭包存在性

是否存在某些任务族 `T*` 与有限局部原语集 `U`，使得：

- `T*` 在 `U` 下可达。
- 求解与纠错不必周期性依赖全局重解释算子 `G`。

如果存在，说明这类任务天然适合局部状态更新式工作方式。

#### 命题 B：闭包不存在性

是否存在某些任务族 `T#`，使得对任意有限局部原语集 `U`：

- 要么任务不可达。
- 要么求解过程中必须周期性使用全局重解释算子 `G`。

如果存在，说明某些任务不可能由纯局部更新方式稳定承载。

#### 命题 C：闭包与修复

给定任务 `tau` 的一个几乎正确但局部出错状态 `s_bad`，是否存在有限局部原语序列，把 `s_bad` 修复到满足 `Goal(tau, s)` 的状态，而不依赖 `G`。

若存在，可称该任务的局部修复在 `U` 下闭包。

### 为什么重要

这个问题比“有没有 load/store”更接近机制层。它直接问：

- 局部状态更新是否足够承载求解与纠错。

如果答案是肯定的：

- 显式状态语义、局部读写、局部修复就不是纯协议层整理。
- 它们接近一种独立工作方式。

如果答案是否定的：

- 再怎么设计局部协议，也仍要频繁退回全局重解释。

### 与现实系统的关系

现实中的强 LLM+Agent 已经不是纯 append-only。它可以：

- 调用 regex。
- 读写文件。
- 使用数据库。
- 借助检索和外层 memory。

因此今天的问题不是 append-only 与 load/store 是否有理论差异，而是：

- 一个 append-only 模型外接足够强工具和 scaffold 后，在多大程度上已经模拟 load/store。
- 这种模拟的代价、粒度、稳定性、可训练性是什么。

### 与图灵机框架的关系

从可计算性角度，局部状态更新系统当然可纳入图灵机或等价计算模型框架；它不是在挑战图灵完备性。

重点不在“能不能模拟”，而在：

- 哪种受限或增强计算模型更贴近现实智能系统中的高效工作方式。
- 模拟发生在什么层级。
- 模拟代价、状态粒度与控制复杂度如何变化。

### 与 Candidate A/B/C 的关系

- Candidate C 问事件对象是否形成、是否可归因、是否可纠偏，是现实落地入口。
- Candidate A 就是本页的理论化版本，问局部更新是否形成闭包。
- Candidate B 要在闭包问题清楚之后，继续问闭包是否降低修复成本与控制步数。

### 当前缺口

必须尽快补上：

- 对“局部性”的严格定义。
- 对 `G` 的行为判定标准。
- “近似闭包”的容许边界。
- 至少一个现实任务族到 `S/U/G` 的映射例子。

### 最小推进要求

下一步不能先追求一般定义，而应：

1. 先选具体任务族。
2. 先给出 `S / U / G` 映射。
3. 先定义最弱可判定版闭包。

相关 challenge / defense 见 [[10-控制反馈-TokenInstruction/理论与Defense#控制反馈：理论问题 Challenge 与 Defense|理论问题Challenge与Defense]]。

## 控制反馈：理论问题 Challenge 与 Defense

这里整理“理论问题与候选桥”这组文档中的反方挑战与 defense，核心作用是继续收缩为什么押 `局部状态更新闭包`。

> [!summary] 防守边界
> 理论问题变多不等于发散，但必须明确 `primary bet / secondary support / downstream consequence`。当前首要赌注仍是 [[10-控制反馈-TokenInstruction/理论与Defense#局部状态更新闭包|局部状态更新闭包]]。

### 核心 Challenge

理论问题变多、分层变清楚之后，新的风险是：

> 哪一个问题会真正杀出结果，反而更模糊。

当前已区分表示桥、机制桥、现实映射桥、效率桥、内生化桥，以及 Candidate A/B/C。反方会指出：

- 问题太多时，任何一个失败都不伤主线。
- 任何一个成功也未必能定义主线。
- 研究线会显得永远有地方推进，但不容易真正收敛。

### 对 C -> A -> B 顺序的挑战

当前顺序是：

1. Candidate C：先让对象可见、可测。
2. Candidate A：再问是不是机制。
3. Candidate B：最后问有没有效率收益。

反方会指出：

- 这更像研究管理顺序，而不是理论依赖顺序。
- 为什么不能先做闭包问题。
- 为什么不能先做小规模效率下界分析。

Defense：

- 这个顺序确实不是 theorem dependency。
- 它是当前现实系统下更稳的 evidence dependency。
- C 解决对象是否存在、是否可观测；A 解决是不是机制；B 解决机制是否带来效率收益。

### 对局部状态更新闭包的挑战

`局部状态更新闭包` 最有潜力，也最危险。

最大风险：

- `局部性` 尚未严格定义。
- `全局重解释算子 G` 尚未严格定义。
- `近似闭包` 的容许边界尚未定义。

反方判断：

> 这目前更像一个高质量的形式化愿望，而不是已经硬化的形式化对象。

如果 `局部性`、`G`、`近似闭包` 放宽，理论线会吞掉大量时间而不出硬结果。

Defense：

- 不否认这些对象未钉死。
- 当前防守不是“定义已经够硬”，而是“它值得押，但必须 task-first”。
- 先在具体任务族上定义，再看是否能推广。

### 对全局重解释的挑战

挑战：

- “重新解释整个状态”仍像语义隐喻。
- 如何判定它发生。
- 是基于轨迹定义，还是基于系统内部结构定义。
- 两个系统都只看部分上下文时，如何区分全局重解释与局部更新后决策。

Defense：

- 当前 `G` 首先是语义层 / 工作方式层对象，不是 prefill 或 KV cache 的工程指标。
- 但它不能长期停在纯语义层，必须补行为判据。
- 行为判据包括是否重新广泛扫描状态、局部更新是否足以约束下一步、缺少局部约束时是否退回大范围搜索/解释。

### 对纯理论路线的挑战

挑战：

- 理论问题可以很数学，也可以很无关现实。
- 如果最后只证明抽象状态机在某类任务上存在闭包，仍要说明它与今天 LLM+Agent 的映射关系。
- 否则会变成漂亮 toy formalism。

Defense：

- 理论命题必须接受现实映射约束。
- 当前重点不在图灵可模拟性，而在工作方式与复杂度层级。
- 从可计算性看它只是更大模型里的特殊结构；从工作方式看，它可能抓住现实任务的重要结构。

### 对可归因理论化的挑战

挑战：

- 控制动作级错误归因可能只是实验需求的理论包装。
- 如果系统没有漂亮动作级归因但确实高效工作，它是否不算控制反馈。

Defense：

- 可归因的理论地位不如闭包与复杂度 gap 原生。
- 但它是把实验 protocol 反向压成理论对象的入口。
- 它不应成为第一理论主赌注，而应服务实验桥。

### 对局部修复复杂度 gap 的挑战

挑战：

- 它最接近长期效率动机，但极易被任务构造牵着走。
- 任务类可能天然偏向 load/store。
- 成本模型可能默认地址访问优势。
- append-only + external tool protocol 的能力上界可能被低估。

Defense：

- 局部修复复杂度 gap 不应作为第一击。
- 它应作为闭包问题之后的强化。
- 先问是否存在局部机制闭包，再问这个闭包是否带来复杂度收益。

### 对中间状态层的挑战

挑战：

- 中间状态层可能只是过渡态。
- 现实可能走向强 agent/tooling 足够，或轻量 test-time adaptation 足够。

Defense：

- 当前不需要证明中间状态层是最终稳定层。
- 只需要证明在冻结参数、复用现有 LLM 的现实前提下，中间状态层是不可跳过的分析层。

### 当前 Defense 收缩

问题变多不等于发散，前提是角色明确：

- `primary bet`
- `secondary support`
- `downstream consequence`

当前首要赌注应明确押：

- `Candidate A = 局部状态更新闭包`

原因：

- 比 Candidate C 更接近机制层。
- 比 Candidate B 更不容易被脚手架、prompt、工具链污染。
- 比单纯可归因性更接近长期工作方式动机。
- 一旦成立，能反过来重写 C 和 B 的解释框架。

### 强约束

必须立刻收缩到：

- 先选具体任务族。
- 先给出 `S / U / G` 映射。
- 先定义最弱可判定版闭包。

不能先追求最一般定义，不能先谈所有任务，也不能先谈抽象到无法落地的“局部性”。

### 最强诚实结论

- 理论问题很多，但角色不同，不应混成并列主线。
- `局部状态更新闭包` 是当前最值得保留的首要赌注。
- 它必须 task-first 地收缩到 `S / U / G`。
- Candidate C 与 Candidate B 仍重要，但分别更适合作为前置入口与后继强化。
