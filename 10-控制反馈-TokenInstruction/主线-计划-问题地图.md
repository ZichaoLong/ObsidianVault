## 控制反馈：当前主线总览

> [!summary] 当前结论
> 控制反馈线当前不直接赌“新接口已胜出”，而是先押两条可输命题：理论侧的 [[10-控制反馈-TokenInstruction/理论与Defense#局部状态更新闭包|局部状态更新闭包]]，以及实验侧的 [[10-控制反馈-TokenInstruction/实验与机制Defense#控制反馈：事件对象实验计划|事件对象 -> 可归因 -> 可纠偏]]。

### 当前立场

这条线仍值得继续，但理由不是“现有最强 LLM+Agent 完全做不到局部读写”。更准确的判断是：

- 最强 LLM+Agent 已经能借助工具、regex、文件系统、外部 memory、compact/summarize，在某些粒度上模拟 `load/store`。
- 当前实验甚至需要依赖现有 LLM/Agent 作为“智能承载体”，否则很容易退化成纯确定性程序实验。
- 尚未被说明的是：局部读写是否已经形成一层独立、可分离、可分析、可归因、可纠偏的中间状态与控制机制。
- 也尚未说明的是：这层机制在什么任务类上自然、必要，并且可能更高效。

因此当前不直接主张：

- `Token = Instruction` 已被验证。
- `workspace + load/store` 已经全面优于最强 LLM+Agent。
- 它已经构成独立基础模型接口。
- 它已经自然超出 agent engineering。

当前更合理的表述是：

- `Next Token` 仍是基础 bet。
- `Token = Instruction` 是对 token 语义的推广尝试，不是否定 next-token 范式。
- 当前对手不是单一 agent，而是最强 LLM+Agent 已吸收的混合机制族。
- 研究必须先压成可输命题，再谈长期理论与能力收益。

如果这个立场已经足够，直接跳到 [[#控制反馈：当前计划与 Defense|当前计划与 Defense]]。如果想知道它是怎么被一步步压到这里的，往下看 [[#控制反馈：问题展开地图|问题地图]]。

### 双命题结构

当前有两条最小可输命题。

理论主赌注：

> 对某类现实相关任务，求解与纠错是否可以由有限局部状态更新原语持续完成，而不必频繁退回[[#全局重解释|全局重解释]]。

这就是 [[10-控制反馈-TokenInstruction/理论与Defense#局部状态更新闭包|局部状态更新闭包]]。

实验主赌注：

> `workspace + load/store + [[10-控制反馈-TokenInstruction/实验与机制Defense#显式状态语义|显式状态语义]]` 是否能把 `address/read/write/commit` 统一成稳定运行时事件对象，并被后续步骤稳定利用，从而提升错误可归因性与局部纠偏能力。

这就是 [[10-控制反馈-TokenInstruction/实验与机制Defense#控制反馈：事件对象实验计划|事件对象 -> 可归因 -> 可纠偏]]。

两者不是替代关系，而是递进关系：

- 理论命题定义可能存在的工作方式机制。
- 实验命题检验这种机制是否能在现实系统中形成运行时事件对象、可归因结构与可纠偏闭环。
- 若两者站住，再由效率桥追问是否减少局部修复成本、控制步数、搜索成本或全局重解释频率。

### 关键桥梁层

当前最重要的桥梁层不是宏大理论本身，而是：

> `address/read/write/commit` 能否从普通工具调用现象，变成稳定、可回放、可归因、可纠偏、且被后续决策利用的运行时[[10-控制反馈-TokenInstruction/实验与机制Defense#控制反馈：事件对象实验计划|事件对象]]。

这层桥梁的价值在于把长期动机压成现实系统可观测的机制候选。它不是最终理论，也不是能力收益证明。

### 关键限制

理论命题必须接受效率桥裁决。形式上存在局部闭包还不够，最终仍要落回：

- 局部修复成本
- control-step efficiency
- 搜索成本
- 状态更新代价
- 全局重解释频率

实验命题不能停在 diagnostics。事件对象、可归因、可回放只是前置条件，还必须证明：

- 后续步骤是否稳定利用这些对象。
- 是否形成局部纠偏闭环。
- 是否减少对大范围重新理解的依赖。

纯确定性路线与纯智能路线都不够：

- 只做确定性分析，会被降格为算法/复杂度小变体。
- 只做智能实验，会被降格为 tooling / agent engineering 现象。
- 真正增量必须来自：确定性层先钉住机制对象，智能层再证明它是现实有效工作方式候选。

### 当前推进顺序

理论侧：

1. 先押 [[10-控制反馈-TokenInstruction/理论与Defense#局部状态更新闭包|`局部状态更新闭包`]]。
2. 再问局部修复复杂度 gap。
3. 最后触碰更大范围效率桥与层级差异。

实验侧：

1. 先验证运行时事件对象是否形成。
2. 再验证可归因与可纠偏。
3. 再验证是否减少局部修复成本。

### 现实系统定位

“全局重解释”不是实现层概念，不等于是否重新 prefill，也不等于是否重算 KV cache。它是工作方式层概念：下一步决策是否仍主要依赖对大范围状态的再次语义理解。

test-time 参数更新确实是更底层状态机制，但当前阶段先冻结参数，优先验证显式中间状态系统是否已经形成独立工作方式层。

### 长期动机层

这条线最初不是普通 agent engineering，而是从更底层的计算模型与复杂度直觉出发。

长期动机包括：

- `Token = Instruction`：把 token 从纯数据/词扩展到控制动作、读写动作、寻址动作。
- `RAM / RASP` 相比 `append-only / write-once` 模型，可能对应更高效的 test-time computation 组织方式。
- `AI 的时间 Scaling Law`：能力不仅取决于多算几步，也取决于如何组织这些步骤。
- write-once / append-only 模型可能存在效率损失。

这些内容当前只作为理论合法性桥与方向牵引，不是最小命题的直接证据。

如果想先恢复这些出发点本身，再回头看这里，去 [[10-控制反馈-TokenInstruction/历史动机|历史动机]]。

### 现实 Challenge 层

必须承认：

- 强 LLM+Agent 已经能借助工具、regex、文件系统、外部 memory、compact/summarize，在某些粒度上模拟 `load/store`。
- 因此问题不再是“有没有外部 memory”，而是“这种工作方式是否形成独立可分析的机制层”。
- 强 LLM+Agent 不是实验对立面，而是实验必须依赖的智能承载体。

如果此处的追问是“这些机制族到底已经展开到什么程度”，直接看 [[10-控制反馈-TokenInstruction/实验与机制Defense#控制反馈：对标机制谱系|对标机制谱系]]。

### 已展开的关键问题

#### 双命题问题

当前不是单命题，而是双命题：

- 理论侧：机制是否存在。
- 实验侧：机制在现实系统中是否形成可观测、可纠偏、可利用对象。

同时还有一个元层任务：如何把这两个命题定义得足够硬。

如果这里已经把问题空间唤起得差不多了，可以继续看 [[#双命题与 Primary Bet 层级|双命题与 Primary Bet 层级]]；如果想先看现实对手，跳到 [[10-控制反馈-TokenInstruction/实验与机制Defense#控制反馈：对标机制谱系|对标机制谱系]]；如果想直接落回执行，跳到 [[#控制反馈：当前计划与 Defense|当前计划与 Defense]]。

#### Candidate A / B / C

三类候选桥：

- Candidate C：`事件对象 -> 可归因 -> 可纠偏`
- Candidate A：`局部状态更新闭包`
- Candidate B：`局部修复成本 / control-step efficiency`

详见 [[10-控制反馈-TokenInstruction/理论与Defense#控制反馈：理论桥梁与候选命题|理论桥梁与候选命题]] 与 [[10-控制反馈-TokenInstruction/理论与Defense#局部状态更新闭包|局部状态更新闭包]]。

当前判断：

- `A` 是理论主赌注。
- `C` 是现实实验入口。
- `B` 是最终效率桥裁决层。

这不是理论必然顺序，而是当前最稳的证据积累顺序。

#### 全局重解释

“全局重解释”是工作方式层对象，不是实现层对象。它不等于是否重新 prefill，也不等于是否重算 KV cache。

它真正问的是：下一步决策是否仍主要依赖对大范围状态的再次语义理解。

#### 确定性层与智能层

纯确定性路线会被 challenge 成传统算法学或复杂度变体。纯智能路线会被 challenge 成 prompt/tooling/agent engineering 现象。

因此主线必须同时满足：

- 确定性层先钉住机制对象。
- 智能层再证明机制对象会被现实智能系统承载。

#### Test-Time 状态层次

test-time 至少可区分三层：

- token/context 层
- 中间状态系统层
- 参数层

`workspace / load-store` 当前主要研究中间状态系统层。test-time parameter update 是更底层机制，但当前不作为最小命题打开。

详见 [[10-控制反馈-TokenInstruction/实验与机制Defense#确定性层、智能层与 Test-Time 状态层次|确定性层-智能层-状态层次]]。

#### Next Token 与 Token = Instruction：问题地图中的位置

当前不否定 `Next Token`。更准确地说：

- `Next Token` 是已被强力验证、当前不应轻动的基础 bet。
- `Token = Instruction` 是对 token 语义的自然推广。
- 当前是在保持 next-token 训练范式不动的前提下，扩展 token 所承载的语义范围。

详见 [[10-控制反馈-TokenInstruction/主线-计划-问题地图#Next Token 与 Token = Instruction|Next-Token与Token-Instruction]]。

#### NRAM / NTM / DNC

这些历史路线说明显式 memory / addressing 机制是可学的，神经控制器可以读写外部状态。

但它们没有直接回答今天的问题：在强 LLM+Agent 已部分模拟 load/store 的现实前提下，什么仍然是值得独立研究的工作方式增量。

#### 理论与现实映射

核心 challenge 是：即使一个理论对象能被形式化，它是否真的对应现实可实现路径。

若 LLM+Agent 已经图灵完备或足够强，新的机制对象的增量不能是“能不能模拟”，而必须回到代价、粒度、稳定性、可训练性、可诊断性和效率桥。

#### 局部性与更大问题类

[[10-控制反馈-TokenInstruction/理论与Defense#局部状态更新闭包|`局部状态更新闭包`]] 可能只覆盖某些任务族。更大问题类可能天然需要周期性全局操作。

这并不削弱主线，反而更接近复杂度视角下的层级结构：

- 某些任务局部闭包成立。
- 更大任务类需要间歇性全局重解释。
- 再更大任务类可能需要更强状态层或更强全局操作。

闭包失败可能来自状态访问机制问题，也可能来自问题分解结构问题。后续必须区分：是 `U/G` 定义不对，还是递归分解 / 子问题化机制不对。

#### 神经网络承载与学习

长期问题仍然存在：

- 局部状态读写机制最终如何由神经网络承载。
- 是依赖现有 LLM+Agent，还是需要新模型接口、训练接口、甚至 test-time 参数更新。

当前收缩判断：

- 先不同时打开所有学习问题。
- 固定 `Next Token` 作为基座。
- 没有全新神经网络时，实验侧默认只能依赖现有 LLM 作为智能承载体。
- 先在冻结参数前提下验证中间状态层是否值得存在。

## 双命题与 Primary Bet 层级

这页固定当前控制反馈线的双命题结构，以及 `primary bet` 为什么是递进层级，而不是单层命题。

> [!summary] 核心判断
> 当前不是单命题，而是理论命题与实验命题并行推进；理论命题先押 [[10-控制反馈-TokenInstruction/理论与Defense#局部状态更新闭包|局部状态更新闭包]]，但最终仍要接受效率桥裁决。

### 当前两条可输命题

理论侧命题：

> 对某类现实相关任务，求解与纠错是否可以由有限局部状态更新原语持续完成，而不必频繁退回全局重解释。

这就是 [[10-控制反馈-TokenInstruction/理论与Defense#局部状态更新闭包|局部状态更新闭包]]。

实验侧命题：

> `workspace + load/store + 显式状态语义` 是否能把 `address/read/write/commit` 统一成稳定运行时事件对象，并被后续步骤稳定利用，从而提升错误可归因性与局部纠偏能力。

这对应 [[10-控制反馈-TokenInstruction/实验与机制Defense#控制反馈：事件对象实验计划|事件对象实验计划]]。

### 为什么是双命题

当前研究结构至少分成两层：

- 理论命题：工作方式层机制是否存在。
- 实验命题：这层机制在现实系统中是否形成可观测、可纠偏、可验证对象。

因此更准确的说法不是“只剩一个最小命题”，而是：

- 已经明确两个最小可输命题。
- 但两条命题的精确定义、边界与失败条件仍未完全钉死。

### 理论命题不能脱离效率桥

即使 `局部状态更新闭包` 在形式上成立，它若不能回到以下效率桥对象，现实研究意义会变弱：

- 局部修复成本。
- control-step efficiency。
- 搜索成本。
- 状态更新代价。
- 全局重解释频率。

因此理论侧判断应当是：

> `局部状态更新闭包` 是理论主赌注，但它只是前置层，最终必须接受效率桥裁决。

### 两条命题关系

可以理解成递进：

1. 理论命题先定义可能存在的工作方式机制。
2. 实验命题再看这种机制是否在现实系统中形成运行时事件对象、可归因、可纠偏。
3. 若理论站住，再由效率桥追问是否减少局部修复成本。

### 定义命题本身也是最小计划

当前存在一个更小的元层任务：

> 如何把理论命题与实验命题都定义成真正可输、可判定、可继续推进的对象。

这意味着当前最小计划不只是验证命题，也包括先把命题定义收缩到足够硬。

### Primary Bet：第一层

当前最值得押的第一层理论命题是：

> 存在一类现实相关任务，其求解与纠错在有限局部状态更新原语下闭包，而不必频繁退回全局重解释。

关键词：

- 现实相关任务。
- 有限局部状态更新原语。
- 闭包。
- 不必频繁退回全局重解释。

### 为什么第一层足够重要

第一层并不声称：

- 局部性永远最好。
- 全局机制不再需要。
- 所有任务都适合局部闭包。

它只证明：

- 至少有一批现实任务具有局部闭包结构。

只要这一步成立，就足以说明：

- 这条路线不是空的理论重述。
- 围绕显式状态语义、局部状态更新、load/store 风格接口开展实操研究有必要。

### Primary Bet：第二层

更强一步：

> 还存在另一类更大的任务，其求解无法在同样局部原语下闭包，必须引入周期性全局操作。

这把“局部性有用”与“局部性有限”纳入同一个层级图景：

- 不是局部性与全局性二选一。
- 任务类可能呈现局部 -> 混合 -> 全局的层级。

### Primary Bet：第三层

再进一步：

> 这种局部/全局层级差异，是否对应可证明的修复成本、控制步复杂度或 test-time efficiency gap。

这一层接近：

- 长期复杂度动机。
- RAM / RASP vs append-only 的效率直觉。
- Candidate B 的效率桥问题。

### 对主流 LLM+Agent 的启发

如果第一层成立，即使复用现有 Transformer / Attention / KV cache，也会自然诱导出：

- 是否可以减少对大范围全局 token 状态的再次语义理解。
- 是否可以用更小、更局部的状态驱动下一步决策。
- 是否可以把 KV cache 从默认全局状态降级为可选全局回退机制。
- 是否可以引入更明确的中间状态系统，而不是只靠大上下文。

### 对更一般网络架构的启发

如果第一层成立，还会自然继续追问：

- 是否一定需要全稠密 Transformer / Attention。
- 是否存在更一般、更局部通信、更硬件友好的网络架构。
- 是否可以让网络内部状态更接近“局部状态更新 + 稀疏读写”。

这与 TIDE 研究线存在概念连接，但执行上应保持分线。

## 控制反馈：当前计划与 Defense

这页合并当前计划与当前 defense，避免在执行层重复维护两套相近表述。它是从 [[#控制反馈：问题展开地图|问题地图]] 压回来的执行版；如果要恢复来时路，回看 [[10-控制反馈-TokenInstruction/历史动机|历史动机]]。

> [!important] 执行边界
> 当前阶段只推进两条可输命题：[[10-控制反馈-TokenInstruction/理论与Defense#局部状态更新闭包|局部状态更新闭包]] 与 [[10-控制反馈-TokenInstruction/实验与机制Defense#控制反馈：事件对象实验计划|事件对象 -> 可归因 -> 可纠偏]]。任务成功率、完整 test-time training、能力边界和基础模型接口都不是第一击。

当前还不能说：

- 控制反馈理论已经成立。
- `workspace + load/store` 已经全面优于最强 LLM+Agent。
- 它已经构成独立基础模型接口。
- 它已经自然超出 agent engineering。
- 任务成功率提升已经被证明。

当前能守住的是：

- 长期理论动机是严肃的，包括计算复杂度、`RAM/RASP` vs `write-once/append-only`、test-time scaling、CoT 与时序依赖。
- 现实系统没有消化掉全部问题。最强 LLM+Agent 虽已部分模拟 load/store，并吸收 reflection、planning、tool use、search 等机制族，但仍未说明哪一层是独立工作方式层。
- 因而值得把问题压成两条可输命题：理论侧的 [[10-控制反馈-TokenInstruction/理论与Defense#局部状态更新闭包|`局部状态更新闭包`]]，实验侧的 [[10-控制反馈-TokenInstruction/实验与机制Defense#控制反馈：事件对象实验计划|`事件对象 -> 可归因 -> 可纠偏`]]。

如果你此时想先确认“现实系统到底已经吸收了哪些机制族”，插出去看 [[10-控制反馈-TokenInstruction/实验与机制Defense#控制反馈：对标机制谱系|对标机制谱系]]，再回到这里即可。

### 理论计划

第一优先级是 [[10-控制反馈-TokenInstruction/理论与Defense#局部状态更新闭包|局部状态更新闭包]]。当前不再继续扩展问题清单，而要尽快把它压到：

- 一个具体任务族
- 状态空间 `S`
- 局部原语集 `U`
- 全局重解释算子 `G`

这要求 `task-first`，而不是 `definition-first`。先在具体任务族上定义 [[10-控制反馈-TokenInstruction/理论与Defense#最小对象|`S/U/G`]]，再讨论是否能一般化。

第二优先级是在闭包问题有清晰实例后继续追问：

- 局部修复复杂度 gap
- control-step efficiency
- 搜索成本
- 状态更新代价

### 实验计划

第一阶段目标是验证：

- [[10-控制反馈-TokenInstruction/实验与机制Defense#显式状态语义|显式状态语义]] 是否能把 `address/read/write/commit` 统一成稳定运行时事件对象。
- 这些事件是否被后续步骤稳定利用，而不只是日志标签或诊断展示。
- 这种结构是否提升错误可归因性与局部纠偏能力。

实验 protocol 详见 [[10-控制反馈-TokenInstruction/实验与机制Defense#控制反馈：事件对象实验计划|事件对象实验计划]]；Plan-v3 的 challenge / defense 详见 [[10-控制反馈-TokenInstruction/实验与机制Defense#控制反馈：Bridge 与 Plan-v3 Challenge / Defense|Bridge与Plan-v3-Challenge-Defense]]。

主指标：

- 控制动作级错误可归因率。
- 局部纠偏成功率。

强次指标：

- 回放可复现率。
- 同类错误复发率。
- 平均恢复步数。

观察指标：

- 最终任务成功率。
- 状态重建成本。
- 全局重解释触发频率。

### 实验组织

确定性层先固定：

- 状态空间
- 局部原语
- 任务族
- 失败类型

目的不是展示智能，而是钉住机制对象本身是否存在。

智能层再使用现有 LLM/Agent 系统，验证：

- 这种机制是否能被现实预测型控制系统承载。
- 事件对象是否会被模型或系统稳定使用。
- 局部状态读写是否减少全局重解释依赖。

### 当前不作为第一击

当前不把以下方向作为第一主命题：

- 总体任务成功率。
- 大范围 test-time efficiency 曲线。
- 更强能力边界。
- 完整 test-time parameter update。

原因是这些方向太容易被脚手架、prompt、工具链、上下文长度和模型能力污染，容易把机制问题与智能强弱问题混在一起。

### Defense 核心

当前为什么押 [[10-控制反馈-TokenInstruction/理论与Defense#局部状态更新闭包|`局部状态更新闭包`]]：

- 它比事件对象可观测性更接近机制层。
- 它比局部修复复杂度 gap 更不容易被工程因素污染。
- 它比单纯可归因性更接近长期工作方式动机。
- 一旦成立，能反过来重写实验与效率问题的解释框架。

当前为什么押 `事件对象 -> 可归因 -> 可纠偏`：

- 如果 `address/read/write/commit` 不能形成稳定运行时事件对象，并被后续步骤利用，这条路线仍更像规整 protocol 或 diagnostic-aware tooling。
- 当前实验不是绕开最强 LLM+Agent，而是借助它们作为智能承载体，检验显式状态语义与控制接口是否形成可分离机制层。

### 失败条件

当前路线应接受这些失败条件：

- 无法为具体任务族定义有判定力的 [[10-控制反馈-TokenInstruction/理论与Defense#最小对象|`S/U/G`]]。
- 运行时事件对象只是日志标签，不能影响后续决策。
- 可归因与可纠偏无法稳定提升。
- 局部修复成本、控制步数、搜索成本或全局重解释频率没有可解释改善。
- 所谓增量完全可被普通 scaffold / prompt / 工具协议解释。

### 结论

当前 defense 不是“控制反馈已经成立”，而是：这条路线已被压成两个值得押、也能输掉的命题；如果它们站不住，就不应继续扩张叙事；如果它们站住，再由效率桥决定是否值得成为更大的工作方式路线。

## Next Token 与 Token = Instruction

这页固定 `Next Token`、`Token = Instruction`、NRAM/NTM/DNC 与当前控制反馈线的关系。

> [!summary] 核心判断
> `Next Token` 是基础 bet；`Token = Instruction` 是对 token 语义的推广；NRAM/NTM/DNC 说明显式 memory/addressing 机制可学，但没有回答当前强 LLM+Agent 已部分模拟 `load/store` 后还剩什么独立增量。

### Next Token 是基础 bet

当前不应轻易否定 `Next Token` 范式。

原因：

- 它已经在纯语料、自监督、大规模训练中被事实强力验证。
- 它至少证明了：预测下一个 token 是一条非常强的智能逼近路线。

更稳的立场不是：

- `Next Token` 不够本质。

而是：

- `Next Token` 是已被验证的基座。
- 当前要问的是 token 的语义是否可以更广。

### Token = Instruction 是语义推广

`Token = Instruction` 不应被理解成推翻 `Next Token`，而应理解成对纯 `word/data token` 的自然推广。

也就是说，token 不只可以表示：

- 单词。
- 数据。

也可以表示：

- instruction。
- control。
- addressing。
- read。
- write。
- commit。

训练目标仍然可以保持为 `predict next token`，只是 token 序列承载的语义从自然语言扩展到工作方式与控制动作。

### NRAM / NTM / DNC 做了什么

这类路线共同探索：

- 神经控制器。
- 显式外部 memory。
- 可微或半可微 addressing / read / write 机制。

也就是：

- 神经网络 + 显式状态访问。

它们至少提供三类支撑：

- 合法性支撑：显式随机访问 memory、指针操作、外部状态读写不是空想。
- 机制支撑：addressing / read / write 本身可以是神经控制对象。
- 部分现实落地支撑：在算法任务、结构化 memory 任务上，这类机制可以学出来。

### 它们没有证明什么

NRAM / NTM / DNC 没有直接证明：

- 这是通向通用智能的最佳路线。
- 它优于今天的最强 LLM+Agent。
- 它天然适合开放世界长任务。

因此，在当前研究中，它们更像：

- 理论合法性桥。
- 一部分现实落地桥。

而不是最终裁决。

### 对当前路线的启发

NRAM 最重要的启发不是“它已经解决了问题”，而是：

- 显式随机访问 memory 可以被神经系统学出来。
- pointer / addressing 可以成为控制对象。
- `load/store/address generation` 不是凭空想象。

这与 `Token = Instruction` 的直觉一致。

### 当前路线与 NRAM 类路线的差别

起点不同：

- NRAM 类路线更偏显式 memory 神经架构与算法任务。
- 当前路线更偏从 LLM+Agent 现实、计算复杂度与工作方式动机反推。

challenge 不同：

- NRAM 类路线不必面对今天最强 LLM+Agent 已经部分模拟 load/store 的现实。
- 当前路线必须回答：既然最强 agent 已经部分做到局部读写，还剩什么独立问题值得研究。

裁决标准不同：

- 当前路线不能只证明某种 memory 机制可学。
- 还要证明它是否抓住现实任务中的独立工作方式层，以及是否降低局部修复成本或控制复杂度。

### 当前不要同时打开训练范式问题

如果同时质疑：

- `Next Token`。
- token 语义。
- 模型架构。
- test-time training。

变量会过多。

当前更稳的层次：

- 固定不动：暂时接受 `Next Token` 是强基座，复用现有 LLM 是合理起点。
- 当前要动：token 语义、显式状态语义、局部状态更新机制。
- 后续再动：test-time parameter update、更一般的局部通信网络、更远的训练范式变化。

### 结论

`Next Token` 是基础 bet；`Token = Instruction` 是对 token 语义的推广；NRAM/NTM/DNC 说明显式 memory/addressing 机制可学，但没有替代当前路线必须回答的现实问题。
