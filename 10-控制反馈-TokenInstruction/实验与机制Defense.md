# 控制反馈：实验、机制与 Defense

> [!summary] 本页定位
> 本页集中保存事件对象实验计划、对标机制谱系、最小不可替代增量、全局解释算子、确定性层 / 智能层 / 状态层次，以及 Plan-v3 / bridge 压力测试。

## 一页版实验协议

当前实验侧的最小问题不是“整体任务成功率是否更高”，而是：

> `workspace + load/store + 显式状态语义` 是否能把 `address/read/write/commit` 变成稳定运行时事件对象，并据此提升错误可归因性与局部纠偏能力。

第一阶段应固定为：

| 维度 | 最小设定 |
| --- | --- |
| 任务 | JSON / AST / ledger / 依赖图等强结构局部状态任务 |
| 对照组 | A: append-only + standard tools；B: external memory + local tools；C: workspace + read/write + commit/visibility；D: 完整状态语义 |
| 主指标 | 错误类型可归因率 |
| 强次指标 | 局部修复成功率、同类错误复发率、平均恢复步数 |
| 成功条件 | D 相对 C 不只更可观察，还能更稳定触发局部纠偏 |
| 失败条件 | 事件对象不可稳定复现、可归因不提升、纠偏不改善，或收益完全来自脚手架复杂度 |

## 控制反馈：事件对象实验计划

这页是实验侧主入口，对应当前最小实验命题：

> `workspace + load/store + 显式状态语义` 是否能把 `address/read/write/commit` 统一成稳定运行时事件对象，并被后续步骤稳定利用，从而提升错误可归因性与局部纠偏能力。

> [!summary] 第一阶段目标
> 先验证事件对象、回放、归因、局部纠偏是否成立；任务成功率和 test-time efficiency 暂时只作为观察指标。

### 当前定位

第一阶段不直接验证：

- 总体任务成功率是否更高。
- 长任务整体效率是否更优。
- test-time scaling 曲线是否更好。
- 是否已经形成更强基础模型接口。

第一阶段只验证一个前置资格问题：

- 这条路线是否至少形成一个可实验、可重复、可输的工作方式层切面。

### 第一阶段主轴

最小命题：

> 在可回放的局部状态任务上，显式状态语义是否能稳定提升错误类型可归因率，且这种提升不依赖人工后处理。

这个主轴的理由：

- 当前最能守住的不是“能力已经更强”，而是“状态语义是否被显式结构化”。
- 如果结构化不能带来更稳定的错误切分与回放，后续关于训练、诊断、credit assignment 的叙事没有基础。
- 单有可归因还不够，必须进一步支持局部纠偏。

### 优先验证问题

第一阶段优先问：

- `address/read/write/commit` 是否被统一成稳定运行时事件对象。
- 这些事件是否可稳定回放。
- 这些事件是否可稳定归因。
- 这些事件是否被后续步骤稳定利用。
- 这是否提升局部纠偏能力。

### 任务选择

候选任务必须满足：

- 有明确状态对象。
- 允许局部读、局部写、局部提交。
- 有明确 commit / visibility 语义。
- 容易出现错误扩散。
- 可自动判定最终状态是否正确。
- 可人为构造 `address/read/write/commit/update/infer` 类错误。
- 能观测后续步骤是否利用归因结果做局部修复。

第一优先：

- JSON
- AST
- ledger
- 依赖图

第二优先：

- 长代码仓局部修改。
- 长文档分块修订。

选择强结构任务不是为了讨好命题，而是为了控制变量。若第一轮成立，第二轮必须进入更难被解释成 editor/workflow 优势的任务。

### 四组对照

#### A. `append-only + standard tools`

特点：

- 无统一状态对象。
- 无显式提交/可见性语义。
- 无一等回放接口。

作用：

- 提供传统 agent 基线。

#### B. `external memory + local tools`，无统一状态语义

特点：

- 有外部 memory。
- 有局部读写工具。
- 可做 addressing。
- 无统一状态对象。
- 无统一 commit/visibility 规则。

作用：

- 切出“有 memory / 有局部读写”本身的贡献。

#### C. `workspace + read/write + commit/visibility`，无显式回放/诊断接口

特点：

- 有统一状态对象。
- 有明确读写事件。
- 有 commit / visibility 语义。
- 没有把回放/诊断做成一等接口。

作用：

- 切出完整状态语义相对于普通 memory protocol 的贡献。
- 与 D 比较，切出回放/诊断接口本身的贡献。

#### D. 完整状态语义版系统

特点：

- 有统一状态对象。
- 有明确读写事件。
- 有 commit / visibility 语义。
- 有显式回放接口。
- 有显式诊断接口。

作用：

- 作为当前路线第一阶段完整实现。

注意：

- `D > C` 不直接证明控制反馈理论成立。
- 它最多先说明显式回放/诊断接口提供了额外实验价值。

### 指标

主指标：

- 控制动作级错误可归因率。
- 局部纠偏成功率。

配套主轴：

- 回放可复现率。

强次指标：

- 同类错误复发率。
- 平均恢复步数。
- 状态重建成本。

观察指标：

- 最终任务成功率。
- 全局重解释触发频率。

暂缓指标：

- 人工定位错误来源所需时间。

### 错误分类法

第一阶段至少固定：

- `address_error`
- `read_error`
- `write_error`
- `commit_error`
- `update_error`
- `infer_error`

要求：

- 每类错误都能通过任务构造单独诱发。
- 每类错误都能通过轨迹协议自动或半自动识别。
- 不允许把所有失败重新并回“模型答错了”。
- 人工解释只处理边界样本，不能让主指标变成人工阅读轨迹产物。

### 消融

至少做：

- 去掉统一状态对象。
- 去掉 `commit / visibility`。
- 去掉显式回放接口。
- 去掉显式诊断接口。
- 保留局部读写，但退回普通工具协议。
- 保持模型、任务、prompt 尽量不变，只改状态语义层。

### 成功条件

第一阶段成立至少需要：

- D 相比 A、B，在错误类型可归因率上稳定更高。
- D 相比 C，在回放可复现率、归因稳定性或局部纠偏上有增益。
- 增益不能主要依赖人工后处理。
- 增益不能主要由更长上下文、更强 prompt、额外脚手架增强解释。
- 后续步骤确实利用事件对象做局部修复。

### 失败条件

若出现下列情况之一，第一阶段失败或不足以扩展叙事：

- 事件对象只存在于定义中，不存在于稳定运行时轨迹中。
- D 只是在日志层面更整齐，错误归因率没有明显提升。
- D 相比 C 没有明显增益，回放/诊断接口价值不足。
- 归因高度依赖人工解释。
- 回放只是展示层，不能支撑后续局部修复。
- 纠偏收益不明显，或完全可由额外脚手架解释。
- 任何优势都不能说明状态语义真正进入后续决策更新。

### 第二阶段

只有第一阶段成立，才进入第二阶段：

- 看状态语义收益是否转化为局部任务收益。
- 看任务成功率是否稳定提高。
- 看错误恢复步数是否下降。
- 看训练数据清洗、轨迹重放、credit assignment 是否因此受益。

第一阶段先验证研究收益，第二阶段再验证能力收益。

### 最小交付物

本阶段至少产出：

- 一个最小任务定义。
- 一份四组对照协议。
- 一份错误分类法。
- 一份回放协议。
- 一份失败判定模板。

计划压力测试见 [[10-控制反馈-TokenInstruction/实验与机制Defense#控制反馈：Bridge 与 Plan-v3 Challenge / Defense|Bridge与Plan-v3-Challenge-Defense]]。

## 控制反馈：对标机制谱系

这页回答：控制反馈线如果声称研究“新机制”，真正对标的是什么。

结论是：它不对标某一个 agent、某一篇论文或某个单独技巧，而是对标当前最强 LLM+Agent 已吸收的混合机制族。

> [!summary] 对标对象
> 当前对手不是单一 agent，而是 ReAct、planning、reflection、search、tool use、recursive context management、skill memory、recursive decomposition 等机制族的混合体。

> [!tip] 阅读提示
> 这一节不是背景附录，而是实验对标约束。它帮助确定实验不能只赢弱 baseline，而要能说明相对现有 agent 机制族的独立增量。

### 为什么不是单一 Agent

把对面写成“当前最强 LLM+Agent”仍不够精确，因为前沿 agent 背后已经隐含一整套机制组合。

因此当前路线真正要面对的是：

- reasoning / acting 闭环
- planning / execution 分离
- reflection / self-correction
- search / deliberate inference
- program / tool delegation
- recursive context management
- skill / memory accumulation
- recursive decomposition / problem factorization

### 主要机制族

#### ReAct：交替推理与行动

核心价值：

- 在“想一步”和“做一步”之间交替。
- 把 reasoning 与 acting 串成观察驱动的闭环。

主要改善：

- 工具调用时机。
- 任务推进中的局部决策。

不直接解决：

- 显式状态语义。
- 局部状态更新闭包。
- 可归因事件对象。

#### Plan-Then-Execute：先规划，再执行

代表包括 `Plan-and-Solve`、`ReWOO`、`LLMCompiler`。

核心价值：

- 避免只靠局部贪心推进。
- 先形成较粗全局结构。
- 在某些场景下降低执行混乱与遗漏。

通常不直接回答：

- 状态对象是否成为一等公民。
- 读写与提交是否具有统一语义。

#### Reflection / Self-Correction：失败后反思再修

代表包括 `Reflexion`、`Self-Refine`。

核心价值：

- 失败后不只是重试，而是借助语言反馈形成下一轮修正。

主要改善：

- 试错闭环。
- 自我修正。
- 任务后验改进。

局限：

- memory 通常仍表现为文本反思、经验摘要或 prompt 中的后验补丁。
- 不等于受约束的显式状态层。

#### Search / Deliberate Inference：多分支探索

代表包括 `Tree of Thoughts`、`LATS`。

核心价值：

- 单路径解码不够时，引入 lookahead、分支、回溯、评分。

主要改善：

- 搜索结构。
- 推理时额外计算的组织方式。

不直接等价于：

- 显式状态访问机制。
- 局部状态读写协议。

#### Program / Tool Delegation：外包给程序或工具

代表包括 `Toolformer`、`PAL`、`Program of Thoughts`。

核心价值：

- 模型不必自己完成所有子过程。
- 在可程序化子问题上借助外部执行器。

这是当前先进 agent 已广泛吸收的能力，但它不自动推出：

- 状态语义已经统一。
- 事件对象已经成为稳定运行时对象。

#### Recursive Context Management：递归管理上下文

代表包括 `Recursive Language Models (RLMs)`。

核心价值：

- 长上下文不能总是整块喂入。
- 模型需要组织上下文访问、切片、递归处理。

与当前路线接近之处：

- 都在挑战 `append-only context tax`。

差异：

- RLM 更偏上下文访问策略。
- 当前路线更偏统一状态语义、显式事件对象与局部状态更新闭包。

#### Skill / Memory Accumulation：技能与经验沉淀

代表包括 `Voyager`。

核心价值：

- 跨回合、跨任务积累经验。
- 形成外部技能库与可复用经验。

差异：

- 它更像经验外化与检索增强。
- 当前路线关注更细粒度状态读写协议。

#### Recursive Decomposition：递归分解与问题分解

递归分解不应简单并入普通 planning。它回答的不是“下一步先做什么”，而是：

- 原问题如何被改写成若干子问题。
- 子问题之间如何组合回原问题。
- 哪些局部结果可以形成全局解的一部分。

启发来源包括：

- 递归算法。
- 分治。
- 数学证明中的引理分解。
- `lambda calculus` 式组合与展开。
- 问题本身的可分解性结构。

它与当前路线是正交维度：

- `Token = Instruction` 更偏状态访问与控制原语。
- 递归分解更偏问题结构的改写与组合。

### 当前路线与这些机制的关系

控制反馈不应被理解为替代 planning、reflection、search 或 tool use。

更合理的理解是：

- 这些机制族已经部分内化到最强 agent 中。
- 它们构成现实系统默认对手。
- 当前要问的是：在这些机制都存在的前提下，还剩下什么机制增量值得独立研究。

可能独立出来的不是“更会想、更会规划、更会反思”，而是：

- 更显式的状态语义。
- 更受约束的读写与提交机制。
- 更稳定的运行时事件对象。
- 更局部的错误修补闭环。

### 目标效果

当前不应把目标写成泛泛的“更强 agent”或“更好长任务”。更精确的目标是：

- 局部闭环：在某类任务上主要通过局部状态修补推进，而不是频繁全局扫描与重解释。
- 状态语义：中间状态是显式对象，可定义读写、提交与可见性。
- 可归因 / 可纠偏：失败能稳定切成 `address/read/write/commit/infer`，并据此局部修复。
- 效率桥：最终裁决对象是局部修复成本、control-step 数量、搜索成本和全局重解释频率。

还必须跟踪一个维度：问题分解结构是否影响局部状态更新闭包。

### 反向约束

- 如果方案只是把 reflection / plan-execute / search / tool-use 再组合，很容易被吸收。
- 如果方案不能说明状态语义与局部闭环的独立增量，很难成为主线贡献。
- 如果方案最终不能落到局部修复成本、control-step efficiency 等效率桥对象上，很难从机制候选走向工作方式路线。
- 如果方案混淆“问题分解失败”和“状态机制失败”，就很难解释闭包失败发生在哪一层。

## 控制反馈：最小不可替代增量与全局解释算子

这页固定两个问题：

- 控制反馈线到底想在哪个最小不可替代增量上站住。
- `全局解释算子 G` 在不重新发明神经网络的前提下如何现实承载。

> [!summary] 最小增量
> 当前要站住的不是“更强 agent”，而是：局部状态读写是否已从普通工具使用现象，上升为一层独立工作方式机制。

> [!tip] 阅读提示
> 这一节解释实验为什么不是单纯换工具集。它服务于实验设计中的 C/D 对照组边界。

### 不应站在“更强 Agent”上

如果目标写成更强 agent、更会规划、更会反思、更会用工具，这条线会很快被当前最强 LLM+Agent 的混合机制族吸收。

因此当前不应主张：

- 我们也能做 reflection。
- 我们也能做 planning。
- 我们也能做 tool use。

当前要站住的是一个更小、更硬的问题。

### 最小不可替代增量

当前最合理的表述是：

> 局部状态读写，是否已经从普通工具使用现象，上升为一层独立的工作方式机制。

这需要四个条件。

#### 显式状态语义

系统是否真的拥有一层中间状态对象，而不只是：

- 文本堆积。
- prompt 补丁。
- 文件修改痕迹。
- 事后日志。

#### 受约束的控制动作

这层状态是否支持受约束的：

- `address`
- `read`
- `write`
- `commit`

而不是只有笼统的“调一个工具”。

#### 事件进入后续决策

这些事件是否被后续步骤稳定利用，而不是：

- 只被记录。
- 只可回放。
- 只用于诊断展示。

#### 局部修补闭环

系统是否因此更稳定地形成：

- 局部修补。
- 局部纠偏。
- 局部继续推进。

而不必频繁退回大范围重新理解。

### 为什么这是不可替代增量

强 LLM+Agent 已经能做很多局部修改，所以今天不问：

- 会不会改一行文件。
- 会不会只读取一小段。
- 会不会做局部 patch。

真正要问的是：

- 这些局部动作是否对应一层可分离、可分析、可归因、可纠偏的工作方式层。

如果答案是否定的，当前路线仍有独立价值。若答案是肯定的，这条路线更像对已有 agent 机制的描述重写。

### 真正分界线

当前路线若想站住，真正要守的是机制更可分解：

- 错误能稳定切成 `address/read/write/commit/infer`。
- 修复能局部发生。
- 后续步骤显式依赖这些局部状态对象。
- 轨迹因此更适合回放、纠偏与后续训练。

如果这些不成立，路线会被降格为：

- 更规整的 scaffold。
- 更细的工具协议。
- 更好的工程实践。

### `G` 不应先定义为新神经网络

`全局解释算子 G` 当前不应先被定义成：

- 新神经网络模块。
- 底层计算图变化。
- 是否重新 prefill。
- 是否重算 KV cache。

当前阶段，`G` 更适合先定义成行为层算子。

### `G` 的现实定义

可以先把 `G` 定义为：

> 当系统不足以仅凭当前局部状态对象继续推进时，触发一次对更大范围任务状态的重新组织、重新摘要、重新检索、重新规划或重新解释。

关键点：

- `G` 不是“是否重跑模型”的实现问题。
- `G` 是“下一步是否要重新依赖更大范围语义理解”的工作方式问题。

### 现有 LLM+Agent 中 `G` 的承载

在不重新发明神经网络的前提下，`G` 可以由现有系统中的这些操作承载：

- 重新扫描相关文件或更大工作区。
- 重新生成全局计划。
- 重新做 compact / summarize。
- 重新构造 task state / working memory。
- 重新选择接下来关注的对象集合。

因此当前的 `G` 不是新架构，而是现有 LLM 对更大范围状态做一次高层重解释。

### `U` 与 `G` 的现实区分

`U` 是局部更新原语：

- 只改动小状态对象。
- 后续步骤主要依赖这些局部对象继续推进。
- 不需要重新组织大范围任务语义。

`G` 是全局解释算子：

- 必须重新看更大范围状态。
- 重新组织工作记忆、计划或关注区域。
- 后续步骤重新建立对任务整体的把握。

### 这样定义的价值

- 不依赖先造新模型，实验仍可依赖现有 LLM 承载。
- 理论侧可以继续问某类任务是否存在 `U`-闭包，以及能否延后或减少触发 `G`。
- 实验侧可以统计 `G` 的触发频率、触发条件、任务推进中对 `G` 的依赖度，以及局部修补能否替代部分全局解释。

### 结论

当前最该站住的是：

> 局部状态读写是否能成为一层独立工作方式，而不只是普通工具现象。

当前最现实的 `G` 是：

> 现有 LLM+Agent 上的“大范围重新组织与重新解释”行为算子。

## 确定性层、智能层与 Test-Time 状态层次

这页固定两个判断：

- 控制反馈线不能只靠纯确定性路线，也不能只靠纯智能实验路线。
- test-time 状态可分层，参数更新是更底层机制，但不应作为前期最小命题。

> [!summary] 双重要求
> 确定性层负责钉住机制对象，智能层负责证明机制对象不是数学玩具；前期实验先冻结参数，只研究显式状态语义与 workspace。

> [!tip] 阅读提示
> 这一节用于约束实验范围：前期不要打开 test-time training，也不要退化成纯确定性程序实验。

### 纯确定性路线为什么不够

如果只做：

- 固定状态空间。
- 固定局部原语。
- 固定任务。
- 再做复杂度分析或存在性分析。

会被合理 challenge：

- 这不就是已有算法复杂度/计算机科学问题的变体吗？

因此，纯确定性路线单独的研究价值有限。

### 纯智能实验路线为什么不够

如果只做：

- 现有 LLM。
- 外加工具协议。
- 观察某些任务表现。

会被合理 challenge：

- 这只是 prompt / tooling / agent engineering。
- 没有证明背后有独立机制对象。

因此，纯智能实验路线单独也不足以支撑主线贡献。

### 确定性层的正确角色

确定性层不是为了重新发明算法学，而是作为：

- 变量隔离器：把工作方式结构与智能强弱、学习能力、prompt、脚手架变量拆开。
- 机制对象发生器：产生局部闭包任务族、非闭包任务族、局部修复成本对象、全局重解释需求对象。
- 现实实验支架：提供最小状态空间、最小原语集、最小任务族、最小失败条件。

### 智能层的正确角色

智能层不是为了简单让系统做成任务，而是回答：

- 现有预测型控制系统是否会承载这种机制。
- 承载后是否减少全局重解释。
- 承载后是否降低局部修复成本或控制复杂度。
- 这套机制是否能被学习出来、稳定使用。

智能层要证明确定性层得到的机制对象不是纯数学玩具，而是现实智能系统的有效工作方式候选。

### 两层缺一不可

只有确定性层，会被降格为算法问题或抽象复杂度问题。

只有智能层，会被降格为 prompt/tooling/agent engineering 现象。

真正增量来自：

1. 确定性层先给出机制对象。
2. 智能层再验证该机制对象是否真实进入系统工作方式。

### Test-Time 状态层次

test-time 状态可粗略分成三层：

1. `token / context` 层状态。
2. 中间状态系统，包括 `KV cache`、`RNN state`、`node memory`、`workspace`。
3. 参数层状态，即 test-time 更新后的权重。

如果允许 test-time 参数更新，系统就获得比 context、KV cache、workspace 更底层的状态演化机制。

### 为什么参数更新不是前期最小命题

参数层状态更强，但当前不适合作为最小可输命题：

- 变量过多：难以区分收益来自状态语义、外部 workspace、训练规则还是参数适应本身。
- 现实成本过高：工程复杂、成本高、稳定性差、复现实验难。
- 掩盖基础问题：难以回答单靠显式状态语义与局部状态更新，是否已经能形成独立工作方式层。

### 参数更新的长期意义

未来若允许 test-time 参数更新，理论上可能意味着：

- 系统不只在外部状态上修补，还能把局部规律内化到参数层。
- 某些局部修复策略被参数快速吸收，减少重复全局重解释。
- 系统获得更强长期适应能力，尤其在重复结构、重复错误、重复局部模式上。
- 状态语义出现层次分工：workspace 承载显式中间对象，参数层承载更慢、更抽象的适应。

### 复用现有 LLM 的实验路线

路线 A：冻结参数。

- 验证显式状态语义、事件对象、局部修复闭环本身是否成立。
- 这是当前最小可输路线。

路线 B：轻量级 test-time 状态适应，但不动主模型全参。

- 例如 session adapter、临时 LoRA、小型外部状态网络、可学习事件压缩器。
- 研究中间层状态与更底层适应之间的分工。

路线 C：完整 test-time training。

- 研究参数层状态是否进一步降低局部修复成本或全局重解释频率。
- 当前不适合作为前期最小命题。

### 当前实验主张

如果复用现有 LLM，当前更合理的实验主张是：

1. 先冻结参数。
2. 只研究显式状态语义与 workspace。
3. 先证明局部状态更新、可归因、可纠偏是独立存在的。
4. 之后再讨论是否需要更底层参数层适应。

## 控制反馈：Bridge 与 Plan-v3 Challenge / Defense

这页保存 Plan-v3 与 bridge layer 的反方挑战、防守边界与当前结论。它用于压力测试前文实验协议，不替代实验执行入口。

> [!summary] 防守边界
> Plan-v3 只证明“是否值得继续作为独立实验切面推进”，不证明控制反馈理论成立，也不证明能力已经更强。

> [!tip] 阅读提示
> 这一节是计划压力测试。若读者只想实现实验，可以先读 [[#控制反馈：事件对象实验计划]]；若要判断路线是否站得住，必须回来检查这里的失败条件和反方挑战。

### Plan-v3 的总定位

Plan-v3 不是在证明：

- 控制反馈理论已经成立。
- 它已经比最强 LLM+Agent 更强。
- 它已经跨过工具协议边界。

Plan-v3 的作用是：

- 把路线压成可实验、可重复、可输的命题。
- 把能力收益、脚手架效应、状态语义效应先拆开。
- 验证这条路线是否至少具备继续独立推进的资格。

它是前置资格验证计划，不是最终大命题验证计划。

### 对“可归因率”的挑战与 defense

挑战：

- 可归因率提升不等于接口更强。
- 可归因率提升也不等于能力更强。
- 它可能只是更好的诊断或更规整的 protocol。

Defense：

- 可归因率不是终局价值，而是第一阶段最合理锚点。
- 当前最能守住的是状态语义是否被显式结构化。
- 若结构化不能带来稳定错误切分与回放，训练、诊断、credit assignment 叙事无法继续推进。

必须补强：

- 单有可归因不够。
- 必须证明归因支持更稳定的局部纠偏。

### 对两阶段设计的挑战与 defense

挑战：

- 第一阶段赢、第二阶段输，说明什么。
- 第一阶段输、任务成功率有轻微收益，又如何解释。
- 这可能只是把难题延后。

Defense：

- 两阶段拆分是为了避免一开始把所有变量混在一起。
- 先单独验证可诊断性、可回放性、错误切分，再谈能力外显。

必须补一张分叉解释表：

- 第一阶段赢、第二阶段输：更像诊断/协议研究。
- 第一阶段输、第二阶段赢：更像脚手架增强。
- 两阶段都赢：才有资格继续推进接口命题。

### 对 D 组的挑战与 defense

挑战：

- D 组太重，容易把工程复杂度当作研究价值。
- `D > C` 可能只是更完整 protocol 更可观测、更可调试。

Defense：

- 没有 D，就难以切开完整状态语义贡献与回放/诊断接口贡献。
- D 组重不是问题，解释过头才是问题。

结论护栏：

- `D > C` 只能先支持显式回放/诊断接口提供额外实验价值。
- 不能直接上升为控制反馈理论成立。
- 不能直接上升为更强能力接口成立。

### 对错误分类法的挑战与 defense

挑战：

- `address/read/write/commit/update/infer` 看似硬，但可能依赖人工解释。
- 一旦分类需要大量人工语义判断，主指标硬度下降。

Defense：

- 错误分类法的价值在于逼出可证伪结构。
- 它让失败不再是大筐，状态语义开始进入实验协议。

强约束：

- 第一轮任务必须让大部分错误类别可由协议规则自动或半自动识别。
- 人工解释只处理边界样本。

### 对任务筛选的挑战与 defense

挑战：

- JSON / AST / ledger / 依赖图可能是为命题挑任务。
- 这些任务天然适合 protocol、editor agent、workflow engine。

Defense：

- 第一轮强结构任务是为了控制变量、降低噪声。
- 若第一轮都看不出变量，开放世界只会更难解释。

边界：

- 第一轮允许为切变量而挑任务。
- 第一轮成立后，第二轮必须进入更难被解释成 editor/workflow 优势的任务。

### Bridge Layer 的核心挑战

Bridge layer 最大问题不再是保守，而是：

> 它目前仍更像概念层 / 协议层，还没有充分证明自己是工作方式机制层。

主要攻击点：

- “独立工作方式层对象”本身还没有硬判据。
- “统一成稳定运行时事件对象”可能只是表示与记录方式。
- “被后续步骤稳定利用”还没有被压成硬指标。
- `B vs C`、`C vs D` 的解释仍依赖研究者口径。
- 长期理论动机与当前指标之间还缺硬证据桥。

### Bridge Layer 的防守边界

当前 bridge layer 能守的是：

- 长期理论动机解释了为什么路线值得提出。
- 当前桥梁命题解释了为什么第一阶段应先看运行时事件对象、可归因与可纠偏。
- 如果桥梁层都站不住，就不该继续往更大工作方式优势叙事推进。

当前 bridge layer 不能守的是：

- 它已经证明长期复杂度命题。
- 它已经证明更强能力边界。
- 它已经证明 workspace/load-store 全面优于最强 LLM+Agent。
- 它已经证明超出 agent engineering。

### 长期理论动机与当前计划之间的缺口

长期动机包括：

- RAM / RASP vs append-only / write-once。
- 可能存在 test-time efficiency gap。
- 时间不只是成本，也可能是能力资源。
- `workspace + load/store + address generation` 来自更深层工作方式问题。

挑战：

- 这些动机与“错误类型可归因率 / 回放可复现率”之间的必经桥梁还不够硬。
- 如果桥梁只是“它们都和 workspace/load-store 有关”，不够。

当前更准确的关系：

- 长期理论动机解释“为什么值得想”。
- bridge layer 提供“为什么值得做”的候选落脚点。
- Plan-v3 只提供“是否值得继续独立推进”的前置资格测试。

### 现实 Agent 已部分 RAM-like 后的问题

现实 LLM+Agent 已经能结合 regex、文件系统、数据库、检索与工具链，部分获得 RAM-like 行为。

因此真正命题必须在以下方向中收缩：

- 统一状态语义是否优于零散工具拼接。
- 模型本体是否仍缺乏内生的 load/store-style control。
- 事件对象是否真的约束后续行为，而不是记录层。
- 局部修复是否减少恢复步数、同类错误复发率或全局重解释频率。

### 当前最强诚实结论

Plan-v3 守住的不是“更强能力”，而是：

- 路线第一次拥有单轴、可输的第一阶段命题。
- 路线第一次尝试拆开状态语义、脚手架效应、诊断收益。
- 路线第一次把失败条件写成实验协议的一部分。
- 路线第一次把“是否值得继续独立推进”变成可验证问题。

但它仍未证明：

- 为什么这条线必须单独存在。
- 为什么资格测试和控制反馈理论有强绑定。
- 为什么它不是普通 protocol / tooling 改进的自然延伸。
