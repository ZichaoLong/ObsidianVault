# 控制反馈：理论问题 Challenge 与 Defense

日期：2026-05-26  
迁移整理：2026-06-04

这页整理“理论问题与候选桥”这组文档中的反方挑战与 defense。它的作用是保护主线不发散，同时保留为什么押 [[局部状态更新闭包]] 的理由。

## 核心 Challenge

理论问题变多、分层变清楚之后，新的风险是：

> 哪一个问题会真正杀出结果，反而更模糊。

当前已区分表示桥、机制桥、现实映射桥、效率桥、内生化桥，以及 Candidate A/B/C。反方会指出：

- 问题太多时，任何一个失败都不伤主线。
- 任何一个成功也未必能定义主线。
- 研究线会显得永远有地方推进，但不容易真正收敛。

## 对 C -> A -> B 顺序的挑战

当前顺序是：

1. Candidate C：先让对象可见、可测。
2. Candidate A：再问是不是机制。
3. Candidate B：最后问有没有效率收益。

反方会指出：

- 这更像研究管理顺序，而不是理论依赖顺序。
- 为什么不能先做闭包问题。
- 为什么不能先做小规模效率下界分析。

Defense：

- 这个顺序确实不是 theorem dependency。
- 它是当前现实系统下更稳的 evidence dependency。
- C 解决对象是否存在、是否可观测；A 解决是不是机制；B 解决机制是否带来效率收益。

## 对局部状态更新闭包的挑战

`局部状态更新闭包` 最有潜力，也最危险。

最大风险：

- `局部性` 尚未严格定义。
- `全局重解释算子 G` 尚未严格定义。
- `近似闭包` 的容许边界尚未定义。

反方判断：

> 这目前更像一个高质量的形式化愿望，而不是已经硬化的形式化对象。

如果 `局部性`、`G`、`近似闭包` 放宽，理论线会吞掉大量时间而不出硬结果。

Defense：

- 不否认这些对象未钉死。
- 当前防守不是“定义已经够硬”，而是“它值得押，但必须 task-first”。
- 先在具体任务族上定义，再看是否能推广。

## 对全局重解释的挑战

挑战：

- “重新解释整个状态”仍像语义隐喻。
- 如何判定它发生。
- 是基于轨迹定义，还是基于系统内部结构定义。
- 两个系统都只看部分上下文时，如何区分全局重解释与局部更新后决策。

Defense：

- 当前 `G` 首先是语义层 / 工作方式层对象，不是 prefill 或 KV cache 的工程指标。
- 但它不能长期停在纯语义层，必须补行为判据。
- 行为判据包括是否重新广泛扫描状态、局部更新是否足以约束下一步、缺少局部约束时是否退回大范围搜索/解释。

## 对纯理论路线的挑战

挑战：

- 理论问题可以很数学，也可以很无关现实。
- 如果最后只证明抽象状态机在某类任务上存在闭包，仍要说明它与今天 LLM+Agent 的映射关系。
- 否则会变成漂亮 toy formalism。

Defense：

- 理论命题必须接受现实映射约束。
- 当前重点不在图灵可模拟性，而在工作方式与复杂度层级。
- 从可计算性看它只是更大模型里的特殊结构；从工作方式看，它可能抓住现实任务的重要结构。

## 对可归因理论化的挑战

挑战：

- 控制动作级错误归因可能只是实验需求的理论包装。
- 如果系统没有漂亮动作级归因但确实高效工作，它是否不算控制反馈。

Defense：

- 可归因的理论地位不如闭包与复杂度 gap 原生。
- 但它是把实验 protocol 反向压成理论对象的入口。
- 它不应成为第一理论主赌注，而应服务实验桥。

## 对局部修复复杂度 gap 的挑战

挑战：

- 它最接近长期效率动机，但极易被任务构造牵着走。
- 任务类可能天然偏向 load/store。
- 成本模型可能默认地址访问优势。
- append-only + external tool protocol 的能力上界可能被低估。

Defense：

- 局部修复复杂度 gap 不应作为第一击。
- 它应作为闭包问题之后的强化。
- 先问是否存在局部机制闭包，再问这个闭包是否带来复杂度收益。

## 对中间状态层的挑战

挑战：

- 中间状态层可能只是过渡态。
- 现实可能走向强 agent/tooling 足够，或轻量 test-time adaptation 足够。

Defense：

- 当前不需要证明中间状态层是最终稳定层。
- 只需要证明在冻结参数、复用现有 LLM 的现实前提下，中间状态层是不可跳过的分析层。

## 当前 Defense 收缩

问题变多不等于发散，前提是角色明确：

- `primary bet`
- `secondary support`
- `downstream consequence`

当前首要赌注应明确押：

- `Candidate A = 局部状态更新闭包`

原因：

- 比 Candidate C 更接近机制层。
- 比 Candidate B 更不容易被脚手架、prompt、工具链污染。
- 比单纯可归因性更接近长期工作方式动机。
- 一旦成立，能反过来重写 C 和 B 的解释框架。

## 强约束

必须立刻收缩到：

- 先选具体任务族。
- 先给出 `S / U / G` 映射。
- 先定义最弱可判定版闭包。

不能先追求最一般定义，不能先谈所有任务，也不能先谈抽象到无法落地的“局部性”。

## 最强诚实结论

- 理论问题很多，但角色不同，不应混成并列主线。
- `局部状态更新闭包` 是当前最值得保留的首要赌注。
- 它必须 task-first 地收缩到 `S / U / G`。
- Candidate C 与 Candidate B 仍重要，但分别更适合作为前置入口与后继强化。
