# 控制反馈：问题展开地图

日期：2026-05-27  
迁移整理：2026-06-04

这页不是执行计划，而是 challenge / defense 过程中展开出的关键问题地图。执行层仍只落在两条最小可输命题上：

- 理论命题：`局部状态更新闭包`
- 实验命题：`事件对象 -> 可归因 -> 可纠偏`

## 长期动机层

这条线最初不是普通 agent engineering，而是从更底层的计算模型与复杂度直觉出发。

长期动机包括：

- `Token = Instruction`：把 token 从纯数据/词扩展到控制动作、读写动作、寻址动作。
- `RAM / RASP` 相比 append-only / write-once 模型，可能对应更高效的 test-time computation 组织方式。
- `AI 的时间 Scaling Law`：能力不仅取决于多算几步，也取决于如何组织这些步骤。
- write-once / append-only 模型可能存在效率损失。

这些内容当前只作为理论合法性桥与方向牵引，不是最小命题的直接证据。

## 现实 Challenge 层

必须承认：

- 强 LLM+Agent 已经能借助工具、regex、文件系统、外部 memory、compact/summarize，在某些粒度上模拟 `load/store`。
- 因此问题不再是“有没有外部 memory”，而是“这种工作方式是否形成独立可分析的机制层”。
- 强 LLM+Agent 不是实验对立面，而是实验必须依赖的智能承载体。

## 当前研究结构

最小执行主线：

- 理论命题：`局部状态更新闭包`
- 实验命题：`事件对象 -> 可归因 -> 可纠偏`

约束层：

- 现实映射约束：理论对象必须对应现实可实现路径。
- 效率桥约束：实验对象不能只停在更可观测、更可回放。

展开问题层：

- 保存此前已展开但不立即执行的问题。
- 作为后续 challenge 的入口。
- 防止执行主线过度收缩后遗忘背景。

## 已展开的关键问题

### 双命题问题

当前不是单命题，而是双命题：

- 理论侧：机制是否存在。
- 实验侧：机制在现实系统中是否形成可观测、可纠偏、可利用对象。

同时还有一个元层任务：如何把这两个命题定义得足够硬。

### Candidate A / B / C

三类候选桥：

- Candidate C：`事件对象 -> 可归因 -> 可纠偏`
- Candidate A：`局部状态更新闭包`
- Candidate B：`局部修复成本 / control-step efficiency`

详见 [[理论桥梁与候选命题]] 与 [[局部状态更新闭包]]。

当前判断：

- `A` 是理论主赌注。
- `C` 是现实实验入口。
- `B` 是最终效率桥裁决层。

这不是理论必然顺序，而是当前最稳的证据积累顺序。

### 全局重解释

“全局重解释”是工作方式层对象，不是实现层对象。它不等于是否重新 prefill，也不等于是否重算 KV cache。

它真正问的是：下一步决策是否仍主要依赖对大范围状态的再次语义理解。

### 确定性层与智能层

纯确定性路线会被 challenge 成旧算法学或复杂度变体。纯智能路线会被 challenge 成 prompt/tooling/agent engineering 现象。

因此主线必须同时满足：

- 确定性层先钉住机制对象。
- 智能层再证明机制对象会被现实智能系统承载。

### Test-Time 状态层次

test-time 至少可区分三层：

- token/context 层
- 中间状态系统层
- 参数层

`workspace / load-store` 当前主要研究中间状态系统层。test-time parameter update 是更底层机制，但当前不作为最小命题打开。

### Next Token 与 Token = Instruction

当前不否定 `Next Token`。更准确地说：

- `Next Token` 是已被强力验证、当前不应轻动的基础 bet。
- `Token = Instruction` 是对 token 语义的自然推广。
- 当前是在保持 next-token 训练范式不动的前提下，扩展 token 所承载的语义范围。

### NRAM / NTM / DNC

这些历史路线说明显式 memory / addressing 机制是可学的，神经控制器可以读写外部状态。

但它们没有直接回答今天的问题：在强 LLM+Agent 已部分模拟 load/store 的现实前提下，什么仍然是值得独立研究的工作方式增量。

### 对标机制谱系

当前路线真正的对手不是某一篇论文，而是前沿 LLM+Agent 已吸收的机制族：

- ReAct
- plan-then-execute
- recursive decomposition / problem factorization
- reflection / self-correction
- search / deliberate inference
- program / tool delegation
- recursive context management
- skill / memory accumulation

详见 [[对标机制谱系]]。

### 最小不可替代增量与 G

必须明确：

- 当前路线到底想在哪个最小不可替代增量上站住。
- `全局解释算子 G` 在不发明全新神经网络时如何现实承载。

详见 [[最小不可替代增量与全局解释算子]]。

### 理论与现实映射

核心 challenge 是：即使一个理论对象能被形式化，它是否真的对应现实可实现路径。

若 LLM+Agent 已经图灵完备或足够强，新的机制对象的增量不能是“能不能模拟”，而必须回到代价、粒度、稳定性、可训练性、可诊断性和效率桥。

### 局部性与更大问题类

`局部状态更新闭包` 可能只覆盖某些任务族。更大问题类可能天然需要周期性全局操作。

这并不削弱主线，反而更接近复杂度视角下的层级结构：

- 某些任务局部闭包成立。
- 更大任务类需要间歇性全局重解释。
- 再更大任务类可能需要更强状态层或更强全局操作。

闭包失败可能来自状态访问机制问题，也可能来自问题分解结构问题。后续必须区分：是 `U/G` 定义不对，还是递归分解 / 子问题化机制不对。

### 神经网络承载与学习

长期问题仍然存在：

- 局部状态读写机制最终如何由神经网络承载。
- 是依赖现有 LLM+Agent，还是需要新模型接口、训练接口、甚至 test-time 参数更新。

当前收缩判断：

- 先不同时打开所有学习问题。
- 固定 `Next Token` 作为基座。
- 没有全新神经网络时，实验侧默认只能依赖现有 LLM 作为智能承载体。
- 先在冻结参数前提下验证中间状态层是否值得存在。

## 未彻底钉死的定义

`局部状态更新闭包` 仍需落实到：

- 具体任务族
- 状态空间 `S`
- 局部原语集 `U`
- 全局重解释算子 `G`

运行时事件对象仍需落实到：

- `address/read/write/commit` 的判定标准。
- 事件如何被后续步骤稳定利用。
- 如何区分日志标签与真正闭环对象。

效率桥仍需落实到：

- 局部修复成本。
- control-step efficiency。
- 搜索成本。
- 状态更新代价。
