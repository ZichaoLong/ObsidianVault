# 控制反馈：对标机制谱系

日期：2026-05-27  

这页回答：控制反馈线如果声称研究“新机制”，真正对标的是什么。

结论是：它不对标某一个 agent、某一篇论文或某个单独技巧，而是对标当前最强 LLM+Agent 已吸收的混合机制族。

## 为什么不是单一 Agent

把对面写成“当前最强 LLM+Agent”仍不够精确，因为前沿 agent 背后已经隐含一整套机制组合。

因此当前路线真正要面对的是：

- reasoning / acting 闭环
- planning / execution 分离
- reflection / self-correction
- search / deliberate inference
- program / tool delegation
- recursive context management
- skill / memory accumulation
- recursive decomposition / problem factorization

## 主要机制族

### ReAct：交替推理与行动

核心价值：

- 在“想一步”和“做一步”之间交替。
- 把 reasoning 与 acting 串成观察驱动的闭环。

主要改善：

- 工具调用时机。
- 任务推进中的局部决策。

不直接解决：

- 显式状态语义。
- 局部状态更新闭包。
- 可归因事件对象。

### Plan-Then-Execute：先规划，再执行

代表包括 `Plan-and-Solve`、`ReWOO`、`LLMCompiler`。

核心价值：

- 避免只靠局部贪心推进。
- 先形成较粗全局结构。
- 在某些场景下降低执行混乱与遗漏。

通常不直接回答：

- 状态对象是否成为一等公民。
- 读写与提交是否具有统一语义。

### Reflection / Self-Correction：失败后反思再修

代表包括 `Reflexion`、`Self-Refine`。

核心价值：

- 失败后不只是重试，而是借助语言反馈形成下一轮修正。

主要改善：

- 试错闭环。
- 自我修正。
- 任务后验改进。

局限：

- memory 通常仍表现为文本反思、经验摘要或 prompt 中的后验补丁。
- 不等于受约束的显式状态层。

### Search / Deliberate Inference：多分支探索

代表包括 `Tree of Thoughts`、`LATS`。

核心价值：

- 单路径解码不够时，引入 lookahead、分支、回溯、评分。

主要改善：

- 搜索结构。
- 推理时额外计算的组织方式。

不直接等价于：

- 显式状态访问机制。
- 局部状态读写协议。

### Program / Tool Delegation：外包给程序或工具

代表包括 `Toolformer`、`PAL`、`Program of Thoughts`。

核心价值：

- 模型不必自己完成所有子过程。
- 在可程序化子问题上借助外部执行器。

这是当前先进 agent 已广泛吸收的能力，但它不自动推出：

- 状态语义已经统一。
- 事件对象已经成为稳定运行时对象。

### Recursive Context Management：递归管理上下文

代表包括 `Recursive Language Models (RLMs)`。

核心价值：

- 长上下文不能总是整块喂入。
- 模型需要组织上下文访问、切片、递归处理。

与当前路线接近之处：

- 都在挑战 `append-only context tax`。

差异：

- RLM 更偏上下文访问策略。
- 当前路线更偏统一状态语义、显式事件对象与局部状态更新闭包。

### Skill / Memory Accumulation：技能与经验沉淀

代表包括 `Voyager`。

核心价值：

- 跨回合、跨任务积累经验。
- 形成外部技能库与可复用经验。

差异：

- 它更像经验外化与检索增强。
- 当前路线关注更细粒度状态读写协议。

### Recursive Decomposition：递归分解与问题分解

递归分解不应简单并入普通 planning。它回答的不是“下一步先做什么”，而是：

- 原问题如何被改写成若干子问题。
- 子问题之间如何组合回原问题。
- 哪些局部结果可以形成全局解的一部分。

启发来源包括：

- 递归算法。
- 分治。
- 数学证明中的引理分解。
- `lambda calculus` 式组合与展开。
- 问题本身的可分解性结构。

它与当前路线是正交维度：

- `Token = Instruction` 更偏状态访问与控制原语。
- 递归分解更偏问题结构的改写与组合。

## 当前路线与这些机制的关系

控制反馈不应被理解为替代 planning、reflection、search 或 tool use。

更合理的理解是：

- 这些机制族已经部分内化到最强 agent 中。
- 它们构成现实系统默认对手。
- 当前要问的是：在这些机制都存在的前提下，还剩下什么机制增量值得独立研究。

可能独立出来的不是“更会想、更会规划、更会反思”，而是：

- 更显式的状态语义。
- 更受约束的读写与提交机制。
- 更稳定的运行时事件对象。
- 更局部的错误修补闭环。

## 目标效果

当前不应把目标写成泛泛的“更强 agent”或“更好长任务”。更精确的目标是：

- 局部闭环：在某类任务上主要通过局部状态修补推进，而不是频繁全局扫描与重解释。
- 状态语义：中间状态是显式对象，可定义读写、提交与可见性。
- 可归因 / 可纠偏：失败能稳定切成 `address/read/write/commit/infer`，并据此局部修复。
- 效率桥：最终裁决对象是局部修复成本、control-step 数量、搜索成本和全局重解释频率。

还必须跟踪一个维度：问题分解结构是否影响局部状态更新闭包。

## 反向约束

- 如果方案只是把 reflection / plan-execute / search / tool-use 再组合，很容易被吸收。
- 如果方案不能说明状态语义与局部闭环的独立增量，很难成为主线贡献。
- 如果方案最终不能落到局部修复成本、control-step efficiency 等效率桥对象上，很难从机制候选走向工作方式路线。
- 如果方案混淆“问题分解失败”和“状态机制失败”，就很难解释闭包失败发生在哪一层。
