# 控制反馈：当前主线总览

日期：2026-05-27  
迁移整理：2026-06-04

这页是控制反馈线的当前执行入口。它只承载已经收缩后的主线，不试图展开全部历史动机与 challenge；完整问题空间见 [[问题展开地图]]。

## 当前立场

这条线仍值得继续，但理由不是“现有最强 LLM+Agent 完全做不到局部读写”。更准确的判断是：

- 最强 LLM+Agent 已经能借助工具、regex、文件系统、外部 memory、compact/summarize，在某些粒度上模拟 `load/store`。
- 当前实验甚至需要依赖现有 LLM/Agent 作为“智能承载体”，否则很容易退化成纯确定性程序实验。
- 尚未被说明的是：局部读写是否已经形成一层独立、可分离、可分析、可归因、可纠偏的中间状态与控制机制。
- 也尚未说明的是：这层机制在什么任务类上自然、必要，并且可能更高效。

因此当前不直接主张：

- `Token = Instruction` 已被验证。
- `workspace + load/store` 已经全面优于最强 LLM+Agent。
- 它已经构成独立基础模型接口。
- 它已经自然超出 agent engineering。

当前更合理的表述是：

- `Next Token` 仍是基础 bet。
- `Token = Instruction` 是对 token 语义的推广尝试，不是否定 next-token 范式。
- 当前对手不是单一 agent，而是最强 LLM+Agent 已吸收的混合机制族。
- 研究必须先压成可输命题，再谈长期理论与能力收益。

## 双命题结构

当前有两条最小可输命题。

理论主赌注：

> 对某类现实相关任务，求解与纠错是否可以由有限局部状态更新原语持续完成，而不必频繁退回全局重解释。

这就是 `局部状态更新闭包`。

实验主赌注：

> `workspace + load/store + 显式状态语义` 是否能把 `address/read/write/commit` 统一成稳定运行时事件对象，并被后续步骤稳定利用，从而提升错误可归因性与局部纠偏能力。

这就是 `事件对象 -> 可归因 -> 可纠偏`。

两者不是替代关系，而是递进关系：

- 理论命题定义可能存在的工作方式机制。
- 实验命题检验这种机制是否能在现实系统中形成运行时事件对象、可归因结构与可纠偏闭环。
- 若两者站住，再由效率桥追问是否减少局部修复成本、控制步数、搜索成本或全局重解释频率。

## 关键桥梁层

当前最重要的桥梁层不是宏大理论本身，而是：

> `address/read/write/commit` 能否从普通工具调用现象，变成稳定、可回放、可归因、可纠偏、且被后续决策利用的运行时事件对象。

这层桥梁的价值在于把长期动机压成现实系统可观测的机制候选。它不是最终理论，也不是能力收益证明。

## 关键限制

理论命题必须接受效率桥裁决。形式上存在局部闭包还不够，最终仍要落回：

- 局部修复成本
- control-step efficiency
- 搜索成本
- 状态更新代价
- 全局重解释频率

实验命题不能停在 diagnostics。事件对象、可归因、可回放只是前置条件，还必须证明：

- 后续步骤是否稳定利用这些对象。
- 是否形成局部纠偏闭环。
- 是否减少对大范围重新理解的依赖。

纯确定性路线与纯智能路线都不够：

- 只做确定性分析，会被降格为算法/复杂度小变体。
- 只做智能实验，会被降格为 tooling / agent engineering 现象。
- 真正增量必须来自：确定性层先钉住机制对象，智能层再证明它是现实有效工作方式候选。

## 当前推进顺序

理论侧：

1. 先押 `局部状态更新闭包`。
2. 再问局部修复复杂度 gap。
3. 最后触碰更大范围效率桥与层级差异。

实验侧：

1. 先验证运行时事件对象是否形成。
2. 再验证可归因与可纠偏。
3. 再验证是否减少局部修复成本。

## 现实系统定位

“全局重解释”不是实现层概念，不等于是否重新 prefill，也不等于是否重算 KV cache。它是工作方式层概念：下一步决策是否仍主要依赖对大范围状态的再次语义理解。

test-time 参数更新确实是更底层状态机制，但当前阶段先冻结参数，优先验证显式中间状态系统是否已经形成独立工作方式层。
